# Licensing and provenance

Wikidata structured data is CC0. Every accepted Commons file has a separate license record.

Webwoven's project-authored synthetic smoke-fixture data is distributed under the repository's
MIT license. It is test-only data, not imported Wikidata knowledge, and normal gameplay never
silently falls back to it.

The release playtest pack contains real Wikidata labels, descriptions, and statements under CC0.
Its attribution report records the source and snapshot date. Accepted Commons derivatives are
stored inside the immutable graph bundle; runtime gameplay never fetches them from Wikimedia.

The automatic media allowlist accepts Public Domain, CC0, and complete CC BY or CC BY-SA records
from versions 1.0 through 4.0. Unclear, unsupported, and incomplete copyright records are not
published automatically. Every creator-required license must include both a creator or credit and
a matching Creative Commons license URL. Trademark and other non-copyright notices are preserved
in the attribution ledger instead of being mistaken for copyright-license failures.

Every accepted asset record stores source page, original URL, creator/credit, license and URL,
remote hash, local hash, retrieval time, modifications, and automatic review status. Media is
published only when its entity link, raster type, HTTPS Wikimedia origin, complete attribution,
license allowlist, content hash, and manifest entry all pass. Normal gameplay loads a strict bundle
and fails closed unless every graph entity has a verified local image. Project-authored category
plates remain only as an explicit compatibility fallback for synthetic tests and legacy snapshots.

The full-node July 15 atlas maps all 5,482 graph entities to 5,062 distinct Commons source files,
stored locally by content hash.
Direct Wikidata media and exact Wikipedia lead images are preferred. Exact Commons categories,
structured depicts matches, entity-specific article media, and documented graph-neighbor context
fill the long tail. Contextual fallbacks retain the related entity label in the API and Sources
drawer so a creator portrait or award object is never presented as a literal depiction of an
abstract work or category.
